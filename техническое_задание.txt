    1. Введение
        1.1. Основание для разработки
            1.1.1. Наименование работы: «Десктопное приложение для онлайн-школы по программированию».
            1.1.2. Основанием для разработки является задание на выпускную квалификационную работу.
            1.1.3. Организация, утвердившая договор: Читинский Педагогический колледж.
        1.2. Назначение и область применения
            1.2.1. Назначение разработки — создание настольного клиент-серверного приложения для обучения программированию, включающего каталог курсов, теоретические материалы, практические задания, запуск пользовательского кода и проверку решений на сервере.
            1.2.2. Область применения — учебный процесс в онлайн-школе программирования для студентов и преподавателей.
            1.2.3. Основные функциональные задачи приложения включают:
                1.2.3.1. Регистрацию и авторизацию пользователей с разделением ролей (student, teacher);
                1.2.3.2. Просмотр списка курсов (языков программирования), списка тем и связанных заданий;
                1.2.3.3. Просмотр теоретических материалов по выбранной теме (поддержка HTML-разметки в тексте теории);
                1.2.3.4. Просмотр условия задания и публичного примера ввода/вывода;
                1.2.3.5. Редактирование и запуск пользовательского кода с передачей входных данных;
                1.2.3.6. Автоматическую проверку решения на тестах и возврат количества пройденных тестов;
                1.2.3.7. Отображение прогресса обучения по темам и курсам в профиле пользователя;
                1.2.3.8. Восстановление пароля по секретному восстановительному полю;
                1.2.3.9. Администрирование учебного контента преподавателем (создание курсов, тем, заданий и тестов).

    2. Требования к программе
        2.1. Требования к функциональным характеристикам
            2.1.1. Работа с учетными записями и ролями:
                2.1.1.1. Программа должна поддерживать регистрацию с обязательным указанием логина, пароля, роли и восстановительного поля;
                2.1.1.2. Программа должна поддерживать вход по логину и паролю;
                2.1.1.3. Программа должна поддерживать восстановление пароля по логину, восстановительному полю и новому паролю;
                2.1.1.4. Для авторизованного пользователя должна поддерживаться сессия по bearer-токену с ограниченным временем жизни;
                2.1.1.5. Роль teacher должна предоставлять доступ к панели преподавателя, роль student — к пользовательскому режиму обучения.
            2.1.2. Работа с курсами, темами и заданиями:
                2.1.2.1. Должна быть реализована загрузка списка курсов (языков) с кратким описанием и изображением;
                2.1.2.2. Для выбранного курса должна быть реализована загрузка списка тем;
                2.1.2.3. Для выбранной темы должна быть реализована загрузка теоретического материала и списка заданий;
                2.1.2.4. Для выбранного задания должна быть реализована загрузка текста задания и публичного теста (пример входа/выхода);
                2.1.2.5. Навигация должна обеспечивать последовательный переход: курсы → темы → теория → задание.
            2.1.3. Редактор и выполнение кода:
                2.1.3.1. В клиенте должен быть реализован встроенный редактор текста решения;
                2.1.3.2. Пользователь должен иметь возможность запускать код с произвольным входным набором данных;
                2.1.3.3. Сервер должен компилировать/исполнять код во временной директории и возвращать результат выполнения;
                2.1.3.4. Сервер должен поддерживать языки C++, Python, C# и Go;
                2.1.3.5. При компиляционных ошибках должен возвращаться явный результат «Ошибка компиляции», при ошибках исполнения — соответствующее сообщение.
            2.1.4. Проверка решений и прогресс:
                2.1.4.1. Проверка решения должна выполняться на сервере по набору тестов задания;
                2.1.4.2. Результатом проверки должно быть количество пройденных тестов и общий итог;
                2.1.4.3. При полном прохождении тестов задание должно помечаться как решённое;
                2.1.4.4. При решении всех заданий темы прогресс темы должен устанавливаться в 100%;
                2.1.4.5. В профиле должен отображаться общий прогресс, список завершённых/незавершённых тем и прогресс по курсам.
            2.1.5. Функции преподавателя:
                2.1.5.1. Преподаватель должен иметь возможность создавать курс (название, краткое описание, URL изображения);
                2.1.5.2. Преподаватель должен иметь возможность создавать тему (курс, название темы, текст теории);
                2.1.5.3. Преподаватель должен иметь возможность создавать задание (тема, текст, публичный пример);
                2.1.5.4. Преподаватель должен иметь возможность добавлять дополнительные тесты к заданию;
                2.1.5.5. Доступ к операциям создания контента должен быть ограничен серверной проверкой роли teacher.
        2.2. Требования к надёжности
            2.2.1. Требования к обеспечению надежного функционирования программы:
                2.2.1.1. Все API-методы должны валидировать HTTP-метод запроса и обязательные поля входных данных;
                2.2.1.2. При ошибках валидации или авторизации сервер должен возвращать структурированный JSON-ответ с кодом ошибки;
                2.2.1.3. Приложение должно обрабатывать сетевые и серверные ошибки с выводом текста ошибки пользователю;
                2.2.1.4. Для снижения риска зависаний выполнения кода должны использоваться ограничения по времени компиляции и запуска;
                2.2.1.5. Временные каталоги выполнения должны очищаться после завершения запуска/проверки;
                2.2.1.6. Состояние авторизации должно корректно сбрасываться при выходе из аккаунта.
            2.2.2. Время восстановления после отказа:
                2.2.2.1. При отказе клиентской программы восстановление обеспечивается перезапуском приложения;
                2.2.2.2. При истечении серверной сессии пользователь должен повторно авторизоваться;
                2.2.2.3. При сбое серверной части восстановление зависит от причины и включает перезапуск web/API-сервиса и проверку БД.
            2.2.3. Отказы из-за некорректных действий пользователей системы
                2.2.3.1. При пустом или некорректном вводе (логин/пароль/код/ID) должны возвращаться сообщения об ошибке;
                2.2.3.2. При попытке вызова защищённых методов без токена должен возвращаться отказ авторизации;
                2.2.3.3. При попытке доступа студента к операциям преподавателя должен возвращаться отказ по правам доступа.

    3. Условия эксплуатации
        3.1. Климатические условия эксплуатации
            3.1.1. Рабочая температура: от 10 до 35 градусов Цельсия;
            3.1.2. Рабочая влажность: от 30% до 80%.
        3.2. Требования к квалификации и численности персонала
            3.2.1. Для студента и преподавателя специальных навыков администрирования системы не требуется;
            3.2.2. Для сопровождения серверной части требуется специалист, владеющий PHP, SQL и администрированием Linux-сервера.
        3.3. Требования к составу и параметрам технических средств
            3.3.1. Клиентская ОС: Windows 10+ или Linux с поддержкой Qt-приложений;
            3.3.2. Процессор: Intel Core i3 или эквивалент;
            3.3.3. ОЗУ: не менее 4 ГБ;
            3.3.4. Свободное место на диске: не менее 500 МБ для клиента;
            3.3.5. Постоянное интернет-соединение для работы с API;
            3.3.6. Сервер: Linux/Unix-подобная ОС, PHP с PDO MySQL, СУБД MySQL/MariaDB;
            3.3.7. Для выполнения кода на сервере требуются установленные инструменты: g++, python3, mono (mcs/csc+mono), go.
        3.4. Требования к информационной и программной совместимости
            3.4.1. Требования к информационным структурам и методам решения:
                3.4.1.1. Данные должны храниться в реляционной БД с сущностями пользователей, сессий, языков, тем, заданий, тестов и прогресса;
                3.4.1.2. Обмен между клиентом и сервером должен осуществляться по HTTP(S) с JSON-полями;
                3.4.1.3. Идентификаторы курсов, тем и заданий должны использоваться в API-запросах для выборки контента;
                3.4.1.4. Прогресс должен вычисляться на сервере и передаваться клиенту в агрегированном виде.
            3.4.2. Требования к исходным кодам и языкам программирования:
                3.4.2.1. Клиентская часть реализуется на C++ (Qt) и QML;
                3.4.2.2. Серверная API-часть реализуется на PHP;
                3.4.2.3. SQL-скрипт и структура БД должны быть совместимы с MySQL/MariaDB;
                3.4.2.4. Код должен быть структурирован по модулям: клиентский UI, клиентский API-слой, серверные endpoint-методы.
            3.4.3. Требования к программным средствам, используемым программой:
                3.4.3.1. Клиент использует Qt Quick / Qt Quick Controls / Qt Network;
                3.4.3.2. Сервер использует PHP PDO и веб-сервер с поддержкой запуска PHP-скриптов;
                3.4.3.3. Для выполнения пользовательского кода используются системные компиляторы/интерпретаторы и утилита timeout;
                3.4.3.4. Система должна обеспечивать совместимость протокола API между клиентом и сервером при единых именах endpoint-методов.
            3.4.4. Требования к защите информации и программ:
                3.4.4.1. Пароли и восстановительные данные должны храниться в виде хэшей;
                3.4.4.2. Доступ к защищённым API-методам должен осуществляться только по bearer-токену;
                3.4.4.3. Для выполнения пользовательского кода должны применяться ограничения времени выполнения;
                3.4.4.4. Должно быть реализовано разграничение доступа по ролям и запрет несанкционированных операций.
        3.5. Специальные требования
            3.5.1. Интерфейс должен поддерживать пользовательский сценарий без стандартной системной рамки окна (кастомный заголовок окна);
            3.5.2. Должна поддерживаться стартовая заставка (Splash Screen);
            3.5.3. В профиле должен отображаться детализированный прогресс по темам и курсам;
            3.5.4. Панель преподавателя должна быть доступна только после авторизации учётной записи с ролью teacher;
            3.5.5. В программе должен быть реализован выход из аккаунта с очисткой локального состояния авторизации.

    4. Требования к программной документации
        4.1. Состав программной документации
            4.1.1. Техническое задание;
            4.1.2. Руководство пользователя (студент);
            4.1.3. Краткое руководство преподавателя по созданию учебного контента;
            4.1.4. Описание серверных API-методов и форматов JSON-ответов.

    5. Технико-экономическое обоснование
        5.1. Разработка ориентирована на учебный процесс и не предполагает коммерческого лицензирования на этапе ВКР;
        5.2. Экономическая эффективность выражается в сокращении времени проверки типовых практических задач за счёт автоматизированной проверки;
        5.3. Использование открытых технологий (Qt, PHP, MySQL, компиляторы/интерпретаторы) снижает стоимость внедрения.

    7. Порядок контроля и приемки
        7.1. Предварительный контроль
            7.1.1. Проверка соответствия функциональности приложения утверждённому ТЗ;
            7.1.2. Контроль корректности API-взаимодействия клиента и сервера;
            7.1.3. Контроль корректности разграничения доступа по ролям.
        7.2. Тестирование
            7.2.1. Проверка регистрации, входа, восстановления пароля и выхода из аккаунта;
            7.2.2. Проверка отображения курсов, тем, теории и заданий;
            7.2.3. Проверка запуска пользовательского кода по поддерживаемым языкам;
            7.2.4. Проверка автоматической проверки решений и обновления прогресса;
            7.2.5. Проверка функций преподавателя по созданию контента;
            7.2.6. Проверка обработки ошибок и отказоустойчивости на некорректных входных данных.
        7.3. Приёмка
            7.3.1. Приёмка осуществляется при успешном прохождении функциональных тестов по ролям student и teacher;
            7.3.2. Обязательным условием является корректная работа запуска и проверки решений;
            7.3.3. Обязательным условием является корректный расчёт и отображение прогресса обучения;
            7.3.4. Результаты испытаний оформляются протоколом и утверждаются руководителем проекта.
